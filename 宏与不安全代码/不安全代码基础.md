# 不安全代码基础

> 子文档：本文件涵盖 unsafe 的核心概念——不安全块/函数、契约模型、未定义行为清单、不安全特型。裸指针、内存操作、联合体、FFI 等实操内容请参见：[裸指针与内存操作](裸指针与内存操作.md)。

基于《Rust程序设计（第2版）》第22章。

---

## 1. 不安全因素来自哪里

**不安全特性**是强加了某种**契约**的特性：**Rust 不能自动执行这些规则，但你必须遵守这些规则以避免未定义行为**。

这种契约超出了常规类型检查和生命周期检查的能力范围，仅在该特性的文档中进行解释。当使用不安全特性时，作为程序员，你有责任检查自己的代码是否遵守了它们的契约。

只能在 `unsafe` 块或 `unsafe` 函数中使用不安全特性，这可以避免在不知不觉中使用不安全特性。

---

## 2. 不安全块（`unsafe` 块）

`unsafe` 块看起来就像前面加了 `unsafe` 关键字的普通 Rust 块：

```rust
unsafe {
    String::from_utf8_unchecked(ascii)
}
```

`unsafe` 块解锁了 **5 个额外能力**：

1. **调用 `unsafe` 函数**：每个 `unsafe` 函数都必须根据自己的目的指定自己的契约。
2. **解引用裸指针**：安全代码可以传递裸指针、比较它们，并从引用转换成它们，但只有不安全代码才能真正使用它们来访问内存。
3. **访问 `union` 的各个字段**：编译器无法确定这些字段是否包含其各自类型的有效位模式。
4. **访问可变的 `static` 变量**：Rust 无法确定线程何时使用可变静态变量，因此它们的契约要求你确保所有访问都能正确同步。
5. **访问通过 FFI 声明的函数和变量**：即使声明为不可变的，这些函数和变量也仍然被看作 `unsafe` 的，因为它们对于用其他语言编写的代码仍然是可见的。

> **设计意图**：`unsafe` 块的主要目的不是阻止你做任何事，而是**将人们的视线吸引到 Rust 无法保证其安全性的代码上**，引起评审者的更多关注。

## 3. 示例：高效的 ASCII 字符串类型（Newtype 模式）

使用 `Ascii` 类型说明如何通过 **newtype 模式** 构建安全接口：

```rust
mod my_ascii {
    #[derive(Debug, Eq, PartialEq)]
    pub struct Ascii(
        // 必须只持有格式良好的 ASCII 文本：字节范围从 0 到 0x7f
        Vec<u8>
    );

    impl Ascii {
        pub fn from_bytes(bytes: Vec<u8>) -> Result<Ascii, NotAsciiError> {
            if bytes.iter().any(|&byte| !byte.is_ascii()) {
                return Err(NotAsciiError(bytes));
            }
            Ok(Ascii(bytes))
        }
    }

    #[derive(Debug, Eq, PartialEq)]
    pub struct NotAsciiError(pub Vec<u8>);

    // 安全、高效的转换——使用不安全代码实现
    impl From<Ascii> for String {
        fn from(ascii: Ascii) -> String {
            // 如果此模块没有 bug，这就是安全的：
            // 格式良好的 ASCII 文本必然是格式良好的 UTF-8
            unsafe { String::from_utf8_unchecked(ascii.0) }
        }
    }
}
```

**关键设计原则**：

- 类型本身是 `pub` 的，但其内部字段**不是**公共的。
- 只有该模块可以构造值或引用其元素，模块代码完全控制允许出现或不允许出现的内容。
- 公共构造函数在构造前会仔细检查，公共方法确保不变条件在整个生命周期中得到维护。

> 在机器层面，newtype 及其元素在内存中具有**相同的表示**，构造 newtype 根本不需要任何机器指令——这是**零成本抽象**。

## 4. 不安全函数（`unsafe fn`）

`unsafe` 函数看起来就像前面加了 `unsafe` 关键字的普通函数。其函数体自动被视为 `unsafe` 块。

只能在 `unsafe` 块中调用 `unsafe` 函数。将函数标记为 `unsafe` 会警告其调用者：**为避免未定义行为，该函数具有他们必须满足的契约**。

```rust
impl Ascii {
    /// 从 `bytes` 构造 `Ascii` 值，不检查内容是否为有效的 ASCII
    ///
    /// # 安全性
    ///
    /// 调用者必须确保 `bytes` 只包含 ASCII 字符：各字节都不大于 0x7f。
    /// 否则，其行为就是未定义的。
    pub unsafe fn from_bytes_unchecked(bytes: Vec<u8>) -> Ascii {
        Ascii(bytes)
    }
}
```

**关于 Bug 和不安全代码的两个关键事实**：

1. **在 `unsafe` 块之前发生的 bug 可能会破坏契约**：`unsafe` 块是否导致未定义行为不仅取决于块本身，还取决于为其提供操作目标的代码。
2. **离开 `unsafe` 块后，仍可能出现此处违约的后果**：未定义行为的后果通常并不会发生在 `unsafe` 块内部，可能直到程序执行了很久之后才引发问题。

## 5. 不安全块 vs 不安全函数

> **推荐判断方法**：先对函数做一些判定。

- **如果能正常编译，但仍可能以导致未定义行为的方式滥用函数** → **必须将其标记为 `unsafe`**。正确使用函数的规则是它的契约，契约的存在意味着函数是不安全的。
- **否则，函数就是安全的**：对函数的任何类型良好的调用都不会导致未定义行为。这样的函数**不应该标记为 `unsafe`**。

> 函数是否在函数体中使用了不安全特性**无关紧要**，重要的是**契约存在与否**。

**常见误区**：不要仅仅因为函数体中使用了不安全特性就把安全的函数标记为 `unsafe`。这会：
- 让函数更难使用
- 让期望在某处找到契约说明的读者感到困惑

相反，应该使用 `unsafe` 块，即便整个函数体只有这一个块。

## 6. 未定义行为清单

**未定义行为** = "Rust 坚定地认为你的代码永远不会出现的行为"。

Rust 判断程序是否"遵纪守法"的规则：

1. **程序不得读取未初始化的内存。**
2. **程序不得创建无效的原始值**：
   - 引用、Box 值或 `fn` 指针为空（`null`）
   - `bool` 值非 `0` 且非 `1`
   - `enum` 值具有无效判别值
   - `char` 值无效（如半代用区的 Unicode 码点）
   - `str` 值不是格式良好的 UTF-8
   - 胖指针具有无效虚表或 slice 长度
   - `never` 类型（`!`）的任何值，只能用于不会返回的函数
3. **程序必须遵守引用规则**：任何引用的生命周期都不能超出其引用目标，共享访问是只读访问，可变访问是独占访问。
4. **程序不得对空指针、未正确对齐的指针或悬空指针进行解引用。**
5. **程序不得使用指针访问与此指针关联的分配区之外的内存。**
6. **程序必须没有数据竞争**：当两个线程在没有同步保护的情况下访问同一个内存位置，并且至少有一个访问是写入时。
7. **程序不得对借助外部函数接口进行的跨语言调用进行栈展开。**
8. **程序必须遵守标准库函数的契约。**

> 未使用不安全特性的 Rust 代码只要编译通过就可以保证会遵守所有规则。只有在使用不安全特性时，遵守这些规则才会成为**你的责任**。

## 7. 不安全特型（Unsafe Trait）

**不安全特型**是一种特型，用于表示这里存在某种 **Rust 无法检查也无法强制保障的契约**。实现者必须满足它，以规避未定义行为。

要实现不安全特型，就必须将实现标记为 `unsafe`：

```rust
pub unsafe trait Zeroable {}

unsafe impl Zeroable for u8 {}
unsafe impl Zeroable for i32 {}
unsafe impl Zeroable for usize {}
// 以及所有整数类型
```

**典型示例：`Send` 和 `Sync`**

`std::marker::Send` 和 `std::marker::Sync` 是不安全特型的典型示例：

- **`Send`**：要求实现者能安全地转移给另一个线程
- **`Sync`**：要求实现者能安全地通过共享引用在线程之间共享

它们没有定义任何方法，但有严格的契约。如果为不合适的类型实现了 `Send`，就会使 `Mutex` 在数据竞争中不再安全。

> **重要**：不安全代码不得依赖于普通的、安全的特型在实现上的正确性。即使安全特型实现错误，不安全代码也**不得表现出未定义行为**。

## 关键设计原则总结

| 原则 | 说明 |
|------|------|
| **不安全特性 ≠ 不安全接口** | 使用不安全代码实现安全接口是 Rust 的惯用模式 |
| **newtype 封装** | 用模块边界控制不变条件，公共 API 保证安全性 |
| **契约文档化** | 每个 `unsafe fn` 必须在文档中明确说明其契约 |
| **最小化 unsafe 范围** | 尽量使用 `unsafe` 块而非将整个函数标记为 `unsafe` |
| **Panic 安全性** | 在暂时放松不变条件的代码区域中，确保不会 panic |
| **零成本抽象** | newtype 在内存中与内部类型具有相同的表示 |

> 裸指针、内存操作（`ptr::read`/`write`/`copy`）、联合体、FFI 等具体实操内容见：[裸指针与内存操作](裸指针与内存操作.md)。
