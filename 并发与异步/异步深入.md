# Rust 异步编程深入

> 前置知识：Future 特型、async/await、spawn、异步块等基础概念请先阅读 [异步基础](异步基础.md)。

---

## 一、`Future` 的 `Send` 问题

当使用 `spawn` 时，`Future` 必须是 `Send` 的。以下情况会出问题：

### 1.1 非 Send 值跨越 await

```rust
async fn reluctant() -> String {
    let string = Rc::new("ref-counted string".to_string());
    some_asynchronous_thing().await;  // 错误：Rc跨了await!
    format!("{}", string)
}
```

### 1.2 解决方案

1. **限制非 Send 值的作用域**，使其不跨越 `await`：
```rust
async fn reluctant() -> String {
    let return_value = {
        let string = Rc::new("...".to_string());
        format!("{}", string)
    };  // Rc 在此离开作用域
    some_asynchronous_thing().await;
    return_value
}
```

2. **使用 `Arc` 代替 `Rc`**

3. **泛型错误类型必须加 Send 约束**：
```rust
// 好
type GenericError = Box<dyn Error + Send + Sync + 'static>;
// 差（非Send）
type GenericError = Box<dyn Error>;
```

## 二、长时间运行的计算

### 2.1 `yield_now` — 主动让出线程

```rust
while computation_not_done() {
    // 完成一个中等规模的计算步骤
    async_std::task::yield_now().await;
}
```

### 2.2 `spawn_blocking` — 在独立线程运行

```rust
async fn verify_password(password: &str, hash: &str, key: &str) -> Result<bool> {
    let password = password.to_string();
    let hash = hash.to_string();
    let key = key.to_string();

    async_std::task::spawn_blocking(move || {
        argonautica::Verifier::default()
            .with_hash(hash)
            .with_password(password)
            .with_secret_key(key)
            .verify()
    }).await
}
```

## 三、Stream（异步流）

`Stream` 是迭代器的异步版本：

```rust
trait Stream {
    type Item;
    fn poll_next(self: Pin<&mut Self>, cx: &mut Context<'_>)
        -> Poll<Option<Self::Item>>;
}
```

### 3.1 使用流

```rust
use async_std::prelude::*;

while let Some(item) = stream.next().await {
    // 使用 item
}
```

### 3.2 扩展特型

`Stream` 的实用方法（`next`、`map`、`filter` 等）定义在 `StreamExt` 扩展特型上，需要 `use async_std::prelude::*`。

## 四、`Pin` 与 `Unpin`

### 4.1 问题：自引用 Future

异步函数返回的 `Future` 包含局部变量和它们的借用。第一次轮询后，`Future` **不能再被移动**（否则内部引用会变成悬空指针）。

### 4.2 `Future` 的两个生命阶段

1. **第一阶段（创建后）**：没有借用，可以安全移动
2. **第二阶段（首次轮询后）**：函数体已开始执行，可能持有内部引用，不能移动

### 4.3 `Pin` 的作用

```rust
pub struct Pin<P> { pointer: P }  // pointer 字段不是 pub
```

- `Pin` 是指针类型的包装器，限制指针的用法以确保目标不会被移动
- 必须是 `Pin<&mut Self>` 才能调用 `poll`
- 获得固定指针的方法：`pin!` 宏、`Box::pin()`、`Pin::from(boxed)`

### 4.4 `Unpin` 特型

- 几乎所有类型都自动实现 `Unpin`（异步函数返回的 `Future` 是例外）
- 对于 `Unpin` 类型，`Pin` 没有限制，可以使用 `Pin::new` 和 `Pin::into_inner`
- `Pin<Box<F>>` 总是实现 `Unpin`（即使 `F` 没有），代价是需要堆分配

### 4.5 一般代码中的 Pin

通常不需要处理 `Pin`：`await`、`block_on`、`spawn` 会在内部管理固定。

## 五、异步互斥锁

### 5.1 何时使用 `async_std::sync::Mutex`

- 任务在持有锁时需要等待其他 `Future`（如 I/O）
- 需要在 `spawn` 中使用的锁守卫

```rust
use async_std::sync::Mutex;

pub async fn send(&self, packet: FromServer) -> ChatResult<()> {
    let mut guard = self.0.lock().await;
    utils::send_as_json(&mut *guard, &packet).await?;
    guard.flush().await?;
    Ok(())
}
```

### 5.2 何时使用 `std::sync::Mutex`

- 持有锁的时间很短，不需要等待任何东西
- 更多用于保护纯内存数据结构的操作

两者的选择：**在持有锁期间不需要 `.await` 时，标准库的 `Mutex` 效率更高**。

## 六、执行器实现原理

### 6.1 唤醒器（Waker）

当 `Future` 返回 `Pending` 时，必须保存 `Context` 中的 `Waker` 并在值得再次轮询时调用：

```rust
// 在 poll 中保存唤醒器
self.waker = Some(cx.waker().clone());
return Poll::Pending;

// 稍后，当结果就绪时
if let Some(waker) = self.waker.take() {
    waker.wake();  // 通知执行器
}
```

### 6.2 简单 `block_on` 实现

```rust
fn block_on<F: Future>(future: F) -> F::Output {
    let parker = Parker::new();
    let unparker = parker.unparker().clone();
    let waker = waker_fn(move || unparker.unpark());
    let mut context = Context::from_waker(&waker);

    pin!(future);

    loop {
        match future.as_mut().poll(&mut context) {
            Poll::Ready(value) => return value,
            Poll::Pending => parker.park(),
        }
    }
}
```

## 七、何时使用异步

### 7.1 异步的真正优势

| 指标 | 线程 | 异步任务 |
|------|------|----------|
| 内存占用 | ≥20 KiB | 几百字节 |
| 创建时间 | ~15 μs | ~300 ns |
| 上下文切换 | ~1.7 μs | ~0.2 μs |

### 7.2 适合场景

- 大量并发连接（如聊天服务器、网络路由器）
- 许多相互通信的独立小任务

### 7.3 异步不加速 I/O

异步 I/O 接口并不比同步接口更高效（操作系统做同样的工作，甚至可能需要两次系统调用）。

### 7.4 Rust 的特殊性

不同于 JavaScript/Python，Rust 的线程已经非常安全（无数据竞争），异步代码的主要优势是**资源效率**和**大量并发连接**。
