# Rust 共享可变状态

## 一、概述

当多个线程需要访问并修改共享数据时，Rust 提供了多种同步原语：

| 原语 | 用途 |
|------|------|
| `Mutex<T>` | 互斥锁，独占访问 |
| `RwLock<T>` | 读写锁，多读单写 |
| `Condvar` | 条件变量，等待通知 |
| `AtomicUsize` 等 | 原子化类型，无锁操作 |
| `lazy_static` | 惰性初始化的全局变量 |

所有原语都体现了 Rust 的核心优势：**数据与锁是一体的**，锁在类型内部保护数据。

---

## 二、`Mutex<T>` 互斥锁

### 2.1 创建与使用

```rust
use std::sync::Mutex;

struct FernEmpireApp {
    waiting_list: Mutex<WaitingList>,
    // ...
}

let app = Arc::new(FernEmpireApp {
    waiting_list: Mutex::new(vec![]),
    // ...
});
```

与 C++ 不同，Rust 中**受保护的数据存储在 `Mutex` 内部**。`Mutex` 仅与锁操作有关，如需堆分配需显式使用 `Arc::new`。

### 2.2 `lock()` 方法与 `MutexGuard`

```rust
impl FernEmpireApp {
    fn join_waiting_list(&self, player: PlayerId) {
        let mut guard = self.waiting_list.lock().unwrap();
        // 进入临界区
        guard.push(player);
        if guard.len() == GAME_SIZE {
            let players = guard.split_off(0);
            drop(guard);  // 提前释放锁
            self.start_game(players);
        }
    }
    // guard 在作用域结束时自动释放锁
}
```

关键点：
- `.lock()` 阻塞直到获得互斥锁
- 返回 `MutexGuard<T>`，它是 `&mut T` 的浅层包装（通过隐式解引用直接调用方法）
- 无法在未持有锁时访问 `Mutex` 中的数据
- **锁不是递归的**：同一线程重复 `.lock()` 会导致死锁

### 2.3 `mut` 与互斥锁的关系

```rust
fn join_waiting_list(&self, player: PlayerId)  // &self 而非 &mut self
```

`Mutex` 通过锁机制在**运行时动态地**提供对内部数据的独占（`mut`）访问，即使对 `Mutex` 本身只有共享（非 `mut`）访问。这是**内部可变性**的一种形式（类似 `RefCell`，但支持多线程）。

## 三、`Mutex` 的局限性

### 3.1 竞态条件

虽然 Rust 保证无**数据竞争**，但使用互斥锁的程序仍可能有其他**竞态条件**——程序行为取决于线程间的运行时间差异。

### 3.2 死锁

```rust
let mut guard1 = self.waiting_list.lock().unwrap();
let mut guard2 = self.waiting_list.lock().unwrap();  // 死锁！
```

互斥锁不是递归锁。防止死锁的最佳实践：**保持临界区尽可能小**。

### 3.3 设计影响

互斥锁会鼓励"再添加一个方法"的工作方式，可能导致代码耦合为单体。优先使用通道等更结构化的方法。

## 四、中毒互斥锁（Poisoning）

### 4.1 什么是中毒

如果线程在持有 `Mutex` 期间 panic，Rust 将该 `Mutex` 标记为**已中毒**。此后任何 `lock()` 调用都返回错误（`PoisonError`）。

### 4.2 中毒的意图

中毒保护的不是"未定义行为"，而是**不变条件**。panic 的线程可能在更新数据的部分字段后退出，破坏了数据的一致性。

### 4.3 强制访问中毒锁

```rust
let mut guard = mutex.lock().unwrap_or_else(|poison_err| {
    poison_err.into_inner()  // 获取底层数据，不管中毒
});
```

### 4.4 传播惯例

```rust
mutex.lock().unwrap()   // panic 从持有锁的线程传播到本线程
handle.join().unwrap()  // panic 从子线程传播到父线程
```

## 五、多消费者通道（`Mutex<Receiver<T>>`）

`mpsc` 通道是单消费者的，但可以使用 `Mutex<Receiver<T>>` 实现多消费者：

```rust
pub mod shared_channel {
    use std::sync::{Arc, Mutex};
    use std::sync::mpsc::{channel, Sender, Receiver};

    #[derive(Clone)]
    pub struct SharedReceiver<T>(Arc<Mutex<Receiver<T>>>);

    impl<T> Iterator for SharedReceiver<T> {
        type Item = T;
        fn next(&mut self) -> Option<T> {
            let guard = self.0.lock().unwrap();
            guard.recv().ok()
        }
    }

    pub fn shared_channel<T>() -> (Sender<T>, SharedReceiver<T>) {
        let (sender, receiver) = channel();
        (sender, SharedReceiver(Arc::new(Mutex::new(receiver))))
    }
}
```

泛型嵌套 `Arc<Mutex<Receiver<T>>>` 直接描述了数据的所有权和线程安全策略。

## 六、`RwLock<T>` 读写锁

### 6.1 使用场景

适合"读多写少"的数据（如服务器配置：加载一次、偶尔重载、频繁只读）。

### 6.2 两个方法

| 方法 | 说明 |
|------|------|
| `read()` | 非 `mut` 访问，允许多个线程同时读取 |
| `write()` | 独占 `mut` 访问，等待所有读取者完成 |

```rust
use std::sync::RwLock;

struct FernEmpireApp {
    config: RwLock<AppConfig>,
    // ...
}

// 读取配置
fn mushrooms_enabled(&self) -> bool {
    let config_guard = self.config.read().unwrap();
    config_guard.mushrooms_enabled
}

// 重载配置
fn reload_config(&self) -> io::Result<()> {
    let new_config = AppConfig::load()?;
    let mut config_guard = self.config.write().unwrap();
    *config_guard = new_config;
    Ok(())
}
```

`RwLock` 完美匹配 Rust 借用规则：单写者或多读者。

## 七、`Condvar` 条件变量

### 7.1 用途

线程等待某个条件变为真。`Condvar` 与 `Mutex` 配合使用。

### 7.2 基本用法

```rust
use std::sync::{Arc, Mutex, Condvar};
use std::thread;

let pair = Arc::new((Mutex::new(false), Condvar::new()));
let pair2 = Arc::clone(&pair);

// 工作线程：等待条件
thread::spawn(move || {
    let (lock, cvar) = &*pair2;
    let mut guard = lock.lock().unwrap();
    while !*guard {
        guard = cvar.wait(guard).unwrap();
    }
    // 条件满足，继续工作
});

// 主线程：设置条件并通知
let (lock, cvar) = &*pair;
let mut guard = lock.lock().unwrap();
*guard = true;
cvar.notify_all();
```

### 7.3 `wait` 方法的关键细节

`Condvar::wait` 的特殊签名：
- **按值消耗** `MutexGuard`，返回新的 `MutexGuard`
- 进入等待前**释放互斥锁**
- 被唤醒后**重新获取互斥锁**

### 7.4 `while` 循环惯用法

```rust
while !guard.has_data() {
    guard = self.has_data_condvar.wait(guard).unwrap();
}
```

使用 `while` 而非 `if` 防止**虚假唤醒**（spurious wakeup）。

## 八、原子化类型

### 8.1 可用类型

| 类型 | 对应 |
|------|------|
| `AtomicIsize` | `isize` |
| `AtomicUsize` | `usize` |
| `AtomicI8`~`AtomicI64` | `i8`~`i64` |
| `AtomicU8`~`AtomicU64` | `u8`~`u64` |
| `AtomicBool` | `bool` |
| `AtomicPtr<T>` | `*mut T` |

### 8.2 基本操作

```rust
use std::sync::atomic::{AtomicIsize, Ordering};

let atom = AtomicIsize::new(0);
atom.fetch_add(1, Ordering::SeqCst);
```

- 原子化操作编译为专门的 CPU 指令（如 x86-64: `lock incq`）
- 无系统调用开销
- 不使用锁

### 8.3 `Ordering` 内存排序

- `Ordering::SeqCst`（顺序一致性）：最严格，最安全，性能损失通常很低
- 拿不准时始终使用 `Ordering::SeqCst`

### 8.4 内部可变性

原子化类型通过共享（非 `mut`）引用获取 `self`——它们是内部可变性的另一种形式。

### 8.5 使用示例：中途取消

```rust
use std::sync::Arc;
use std::sync::atomic::{AtomicBool, Ordering};

let cancel_flag = Arc::new(AtomicBool::new(false));
let worker_cancel_flag = cancel_flag.clone();

let worker_handle = thread::spawn(move || {
    for pixel in animation.pixels_mut() {
        render(pixel);
        if worker_cancel_flag.load(Ordering::SeqCst) {
            return None;  // 取消计算
        }
    }
    Some(animation)
});

// 主线程取消
cancel_flag.store(true, Ordering::SeqCst);
worker_handle.join().unwrap();
```

## 九、全局变量

### 9.1 静态变量默认为不可变

```rust
static PACKETS_SERVED: usize = 0;  // 不可变
```

Rust 阻止全局可变状态。`static mut` 可以声明但访问是不安全的。

### 9.2 原子化全局变量

```rust
use std::sync::atomic::AtomicUsize;

static PACKETS_SERVED: AtomicUsize = AtomicUsize::new(0);

// 递增
PACKETS_SERVED.fetch_add(1, Ordering::SeqCst);
```

### 9.3 `Mutex` 全局变量与 `lazy_static`

`Mutex::new()` 本身不是 `const fn`（在 Rust 1.63 之前），需要 `lazy_static`：

```rust
use lazy_static::lazy_static;
use std::sync::Mutex;

lazy_static! {
    static ref HOSTNAME: Mutex<String> = Mutex::new(String::new());
}
```

原理：
- `lazy_static!` 宏使用 `std::sync::Once` 实现一次性初始化
- 首次解引用时执行初始化，后续返回缓存值
- 每次访问有很小的性能成本（原子化加载指令检查是否已初始化）

### 9.4 `const fn` 的适用性

```rust
const fn mono_to_rgba(level: u8) -> Color {
    Color { red: level, green: level, blue: level, alpha: 0xFF }
}

const WHITE: Color = mono_to_rgba(255);
const BLACK: Color = mono_to_rgba(000);
```

`const fn` 限制：不能以类型为泛型参数（只能以生命周期），不能分配内存，不能操作裸指针。
