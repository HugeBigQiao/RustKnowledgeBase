# Rust 线程与通道

## 一、概述

Rust 支持多种并发编程风格：
- **分叉与合并并行**（fork-join）：启动多个线程独立工作，最后等待所有线程完成
- **通道**（channel）：线程间通过单向管道传递值进行通信
- **共享可变状态**：使用互斥锁等原语安全地共享可变数据（参见[共享可变状态](共享可变状态.md)）

---

## 二、线程启动与联结（fork-join）

### 2.1 `thread::spawn` 启动新线程

```rust
use std::thread;

thread::spawn(|| {
    println!("hello from a child thread");
});
```

`spawn` 接受一个 `FnOnce` 闭包，在新线程中运行。返回 `JoinHandle`。

### 2.2 分叉与合并模式

```rust
use std::{thread, io};

fn process_files_in_parallel(filenames: Vec<String>) -> io::Result<()> {
    const NTHREADS: usize = 8;
    let worklists = split_vec_into_chunks(filenames, NTHREADS);

    // 分叉：为每个块启动一个线程
    let mut thread_handles = vec![];
    for worklist in worklists {
        thread_handles.push(
            thread::spawn(move || process_files(worklist))
        );
    }

    // 联结：等待所有线程完成
    for handle in thread_handles {
        handle.join().unwrap()?;
    }

    Ok(())
}
```

**关键点**：`move` 闭包将所有权转移给新线程，开销极低（仅移动 3 个机器字，不会克隆数据）。

### 2.3 分叉与合并的优点

1. **非常简单**：容易实现且不易写错
2. **避免瓶颈**：没有共享资源锁定，线程自由运行
3. **性能模型直观**：理论上 N 个线程接近 N 倍加速
4. **确定性**：只要线程真正隔离，结果总是确定性的，无竞态条件

### 2.4 联结的必要性

Rust 程序在 `main` 返回后立即退出，即使其他线程仍在运行也不会调用析构器。必须在从 `main` 返回前联结关心的线程。

## 三、跨线程错误处理

### 3.1 `JoinHandle::join()` 的两重作用

```rust
handle.join().unwrap()?;
```

1. `join()` 返回 `std::thread::Result`：**如果子线程发生了 panic，返回 Err**
   - Rust 中 panic 是安全的且局限于每个线程
   - 线程边界充当 panic 的防火墙
   - `.unwrap()` 将 panic 从子线程显式传播到父线程

2. `join()` 将子线程的返回值传回父线程
   - 完整类型示例：`std::thread::Result<std::io::Result<()>>`
   - 外层 `thread::Result` 是 spawn/join API 的
   - 内层是应用程序自己的 `Result`

### 3.2 对比其他语言

- Java/C#：子线程异常默认转储到终端并被遗忘
- C++：默认行为是中止进程
- Rust：错误是 `Result` 值（数据），像其他值一样跨线程传递

## 四、跨线程共享不可变数据（Arc）

### 4.1 不允许共享引用的原因

```rust
// 错误！Rust 不知道子线程要运行多长时间
fn process_files_in_parallel(filenames: Vec<String>, glossary: &GigabyteMap) {
    for worklist in worklists {
        spawn(move || process_files(worklist, glossary))  // 错误
    }
}
```

`spawn` 假设最坏情况：子线程可能永远运行。闭包需要 `'static` 生命周期，但引用不满足。

### 4.2 使用 `Arc` 共享不可变数据

```rust
use std::sync::Arc;

fn process_files_in_parallel(filenames: Vec<String>, glossary: Arc<GigabyteMap>) {
    for worklist in worklists {
        let glossary_for_child = glossary.clone();  // 只克隆Arc指针，增加引用计数
        thread_handles.push(
            spawn(move || process_files(worklist, &glossary_for_child))
        );
    }
}
```

- `Arc::new(giga_map)` 将数据移入堆中
- `.clone()` 只复制智能指针（增加引用计数），不复制数据
- 只要有线程持有 `Arc`，数据就保持存活

## 五、rayon 并行库

`rayon` 提供了更高级的分叉与合并 API：

```rust
use rayon::prelude::*;

// "并行做两件事"
let (v1, v2) = rayon::join(fn1, fn2);

// "并行做N件事"
giant_vector.par_iter().for_each(|value| {
    do_thing_with_value(value);
});
```

### 5.1 并行迭代器

```rust
fn process_files_in_parallel(filenames: Vec<String>, glossary: &GigabyteMap) -> io::Result<()> {
    filenames.par_iter()
        .map(|filename| process_file(filename, glossary))
        .reduce_with(|r1, r2| {
            if r1.is_err() { r1 } else { r2 }
        })
        .unwrap_or(Ok(()))
}
```

### 5.2 工作窃取

`rayon` 使用**工作窃取**技术动态平衡线程间负载，比手动预分配工作更高效。`rayon` 还支持跨线程共享引用，因为并行处理会在 `reduce_with` 返回时完成。

## 六、通道（Channel）

**通道**是单向管道，用于将值从一个线程发送到另一个线程——本质上是线程安全队列。

通道的特性：
- 一端发送数据（`Sender`），一端接收数据（`Receiver`）
- 值的所有权从发送线程转移给接收线程
- 发送和接收都是移动而非复制，即使数据很大也极快
- 通道为空时 `recv()` 会阻塞

### 6.1 创建通道与发送值

```rust
use std::sync::mpsc;

let (sender, receiver) = mpsc::channel();

let handle = thread::spawn(move || {
    for filename in documents {
        let text = fs::read_to_string(filename)?;

        if sender.send(text).is_err() {
            break;
        }
    }
    Ok(())
});
```

关键点：
- `mpsc` = **多生产者**、**单消费者**
- `channel()` 返回 `(Sender<T>, Receiver<T>)`
- `send` 失败意味着 `Receiver` 已被丢弃（另一端已挂断）
- 通过 `move` 闭包将 `sender` 转移给工作线程

### 6.2 接收值

两种等效写法：

```rust
// 方式一：while let
while let Ok(text) = receiver.recv() {
    do_something_with(text);
}

// 方式二：for 循环（推荐，因为Receiver可迭代）
for text in receiver {
    do_something_with(text);
}
```

循环会在通道为空且 `Sender` 已被丢弃时正常退出。

### 6.3 管道模式

管道是将并发引入单线程程序的直观方式。每个线程在程序生命周期内持续生成输出。

```rust
fn run_pipeline(documents: Vec<PathBuf>, output_dir: PathBuf) -> io::Result<()> {
    // 启动所有阶段
    let (texts,   h1) = start_file_reader_thread(documents);
    let (pints,   h2) = start_file_indexing_thread(texts);
    let (gallons, h3) = start_in_memory_merge_thread(pints);
    let (files,   h4) = start_index_writer_thread(gallons, &output_dir);
    let result = merge_index_files(files, &output_dir);

    // 等待所有线程结束
    h1.join().unwrap()?;
    h2.join().unwrap();
    h3.join().unwrap();
    h4.join().unwrap()?;
    result
}
```

管道性能受限于最慢阶段的吞吐量（类似工厂装配线）。

## 七、通道的特性与性能

### 7.1 多发送者

`Sender<T>` 实现了 `Clone`，可克隆出多个发送者：

```rust
let (sender, receiver) = mpsc::channel();
let sender2 = sender.clone();  // 第二个发送者
```

`Receiver<T>` 不能被克隆，需要多个接收端时使用 `Mutex<Receiver<T>>`。

### 7.2 三种队列实现

Rust 通道经过精心优化：
1. **一次性队列**：仅发送一个对象时使用，开销最低
2. **长期队列**：发送第二个值后自动切换，最小化内存分配
3. **多发送者队列**：克隆 `Sender` 后回退到线程安全实现（无锁队列）

即使最慢的实现也是无锁的，仅在队列为空时才需要系统调用（接收线程进入休眠）。

### 7.3 同步通道（背压）

```rust
let (sender, receiver) = mpsc::sync_channel(1000);
```

- 创建时指定容量
- 当通道满时，`send` 会阻塞
- 提供**背压**机制，防止发送者远超接收者速度
- 类似 Unix 管道的固定大小队列

## 八、线程安全：`Send` 与 `Sync`

### 8.1 两个核心特型

| 特型 | 含义 |
|------|------|
| `Send` | 可以安全地将值的所有权传给另一个线程 |
| `Sync` | 可以安全地将值的不可变引用传给另一个线程 |

"安全"指没有数据竞争和其他未定义行为。

### 8.2 自动推导

如果结构体或枚举的所有字段都是 `Send`/`Sync`，Rust 会自动为其实现：

```rust
// Vec和String使用了线程安全的分配器，所以Vec<String>是Send
```

- 大多数类型同时实现了 `Send` 和 `Sync`
- `mpsc::Receiver` 是 `Send` 但不是 `Sync`（保证一次只能被一个线程使用）
- `std::rc::Rc<T>` 既不是 `Send` 也不是 `Sync`（非线程安全的引用计数）

### 8.3 Rc 为什么不是 Sync

如果两个线程同时克隆 `Rc`，会导致引用计数数据竞争——造成 use-after-free 或双重释放（未定义行为）。

### 8.4 编译器强制执行

```rust
use std::rc::Rc;
let rc1 = Rc::new("ouch".to_string());
let rc2 = rc1.clone();
thread::spawn(move || { rc2.clone(); });  // 编译错误！
// error: `Rc<String>` cannot be sent between threads safely
```

`Send`/`Sync` 作为函数类型签名中的限界，当生成线程时传入的闭包必须实现 `Send`。

## 九、通道的其他用法

### 9.1 日志记录服务

多个客户端线程通过通道将日志消息发送到日志线程（通过克隆 `Sender`）。

### 9.2 请求-响应模式

请求可以包含一个 `Sender` 作为"回邮信封"，响应线程用它发送回复。

### 9.3 `try_recv()`

使用 `.try_recv()` 方法轮询结果而不阻塞。

---

## 十、并发编程经验总结

- Rust 坚持安全性，引导开发者构建安全、结构化的通信
- 尽可能使用更结构化的方法（分叉与合并、通道），只在必要时使用 `Mutex`
- 分叉与合并程序具有确定性，不会死锁
- 具有非循环数据流的通道管道也不可能死锁
- Rust 编译器促使"知错就改"而非"出问题后再调试数据竞争"
